## Context — read this first

You're given `starter/lib/sms_console.dart`: a single screen that sends an SMS
and shows a monthly cost breakdown, talking to a multi-tenant backend. Also in
this folder: `API-CONTRACT.md`, the backend contract you build against. You do
**not** need our backend — stub it however you think best.

Two things you should know:

1. **The starter file was generated by an AI coding assistant.** It runs. It
   looks like a reasonable first draft. It also leaks credentials, bills the
   wrong amount, crashes on a slow network, and exposes personal data. None of
   that is obvious from reading it quickly — which is the point.

2. **The engineer we hire will be the only Flutter engineer on this project.**
   Nobody will review your pull requests. The patterns you pick — state
   management, folder structure, error handling, how you test — become the
   patterns the project lives with.

So this is not "can you build a Flutter screen." It is: **can you look at
plausible AI output and see what's wrong with it?** We use Claude Code daily and
expect you to. We're testing whether you're the engineer who *reviews* the
agent's output, or the one who *ships* it unread.

## Part 1 — Review (the main event)

Read `starter/lib/sms_console.dart`. Write `REVIEW.md` with **at least 8
findings**.

For each finding:

- **Severity** — Critical / High / Medium / Low, and what a user or the business
  actually loses.
- **Location** — line number.
- **What's wrong** — the mechanism, not a label. "Bad state management" is a
  label. "`setState` runs after `await` without a `mounted` check, so navigating
  away mid-request throws" is a mechanism.
- **How you'd fix it.**

At least **two** must be defects that would cause a real incident — money wrong,
data leaked, or the app breaking on a normal user's phone — not style opinions.

> Look hardest at: what leaves the device, what happens when the network is slow
> or fails, and anything touching money. Then re-read `API-CONTRACT.md` and ask
> whether the code actually honours it.

## Part 2 — Rebuild it properly

Rebuild the screen as you would if you owned it. This is not a cosmetic pass.

**Requirements:**

- A **state management** approach of your choosing (Riverpod, Bloc, whatever) —
  and a defence of the choice in your ADR. Business logic out of widgets.
- A **data layer**: typed models, a repository, no `dynamic` casts at call
  sites. Money handled so that `0.0079 × 3` is exactly right.
- **Real error and loading states.** Slow network, offline, `429` with
  `Retry-After`, `502` from the provider, expired token. The user should never
  see an infinite spinner.
- **Tenant isolation** honoured per the contract. Switching tenants must never
  show one tenant's data under another's header.
- **Secrets** handled the way you'd defend in a security review.

**And it has to be a screen a real customer would use.** This product ships to
end customers, not to engineers:

- **A theme, not hardcoded styling.** Colours, spacing, and typography defined
  once and referenced — not `€${...}` and magic numbers scattered through
  widgets. Light and dark both.
- **At least two reusable components** extracted with a deliberate API — the
  cost row and the send form are the obvious candidates. Show us where you drew
  the component boundary.
- **The states a user actually hits**, designed rather than defaulted: loading,
  empty (no messages yet), error (with a way to recover), and success. An empty
  list should say something, not render nothing.
- **Basic accessibility**: semantic labels on the interactive elements, tap
  targets that a thumb can hit, and text that survives a larger system font size.

Ship at least: **send SMS**, **paginated message history**, **cost breakdown**.

## Part 3 — Prove it

- **Unit tests** for your data/domain layer, including the money arithmetic.
- **Widget tests** for the send flow, covering at least one failure path.
- **One golden test.**
- Tests must fail if the corresponding bug from Part 1 is reintroduced. A fix
  without a test that would have caught it is a claim, not evidence.

## Part 4 — Make it work on more than a phone

The product ships to **Android and iOS first**, then macOS, Web, Windows, Linux,
from one codebase. One codebase does not mean one layout.

- Make the console genuinely usable at a **360 px phone width** *and* at a
  **1400 px desktop width**. Not a stretched phone layout — a layout that uses
  the space it's given. How you do that (breakpoints, `LayoutBuilder`, an
  adaptive shell) is your call, and we'd like to read why in the ADR.
- **Run it on at least two platforms**, one of which is **not** mobile.
  Screenshot both widths in the README.
- Then write a short section on what actually broke, or would break, across
  platforms — not the marketing answer, the real one. Input, fonts, scrolling
  physics, window resizing, and text selection all behave differently once you
  leave the phone.

We are not asking for visual design. We are asking whether one codebase can
serve a phone and a desktop without either feeling like an afterthought.

## Part 5 — Own it

You're the only engineer here. Show us what that means:

- **`README.md`** — how to run, what you changed and why, what you'd do next with
  another week, what you deliberately did **not** do.
- **One ADR** (`docs/adr/0001-*.md`, one page) on your most consequential
  decision — state-management choice is the obvious candidate. Name the
  alternatives you rejected and why.
- **CI** — GitHub Actions running `flutter analyze` and `flutter test`.
- **`AI-USAGE.md`** — see below.

We would rather see three requirements done properly, with the rest scoped and
explained, than everything done carelessly. Say what you cut and why. That
judgment *is* the seniority signal.

## `AI-USAGE.md` — required

Use Claude Code or any AI tool as much as you like. Then tell us the truth about
it:

- Which parts did you generate, and roughly how?
- **Where was the AI wrong?** At least one concrete case: what it produced, why
  it was wrong, what you replaced it with.
- What did you write yourself because you didn't trust the agent with it?

There is no penalty for heavy AI use. There is a penalty for not knowing what
your own code does. "The agent used `double` for money and I moved to minor units
because 0.1 + 0.2 isn't 0.3 in binary floating point" tells us more than a
hand-written app.

**We will ask you to defend any line of your submission in the interview.**
Submit nothing you cannot explain.

## How we score

| Weight | What we look at |
|---|---|
| **30%** | Part 1 — depth of review. The real bugs, or the cosmetic ones? |
| **25%** | Part 2 — architecture, error handling, and whether the money is right |
| **15%** | Part 2 — is this a screen a customer would use? Theme, components, real states, a11y |
| **15%** | Part 3 — do the tests actually protect the code? |
| **15%** | Parts 4–5 — adaptive layout, docs, ADR, honesty |

What earns nothing: a pixel-perfect UI over a broken data layer, `dynamic`
everywhere, a `REVIEW.md` of formatting nits that misses the API key, or tests
that assert `1 == 1`.

Equally: a correct data layer behind a screen nobody would want to use. This
role builds two end-customer apps. Both halves count.
